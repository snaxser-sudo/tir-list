# AGENTS.md

Инструкции для агентов, работающих с этим проектом.

## О проекте

Это статическое web-only приложение для создания и публикации tier list.

Проект состоит из:

- `index.html`
- `styles.css`
- `script.js`
- `README.md`
- `robots.txt`
- `_headers`

Технологии проекта:

- HTML
- CSS
- Vanilla JavaScript
- без backend
- без базы данных
- без сборщика
- без внешних зависимостей

## Основные правила

1. Не меняй код без явного запроса пользователя. Если задача неочевидная, сначала предложи изменение и дождись подтверждения.

2. Не добавляй React, Vue, Flutter, npm-пакеты, сборщики, backend, базы данных или серверное сохранение без отдельного явного разрешения.

3. Сохраняй проект простым: вся логика должна оставаться в `script.js`, стили в `styles.css`, разметка в `index.html`.

4. Состояние tier list должно храниться в едином объекте состояния. UI должен только отображать это состояние и отправлять пользовательские действия в функции изменения состояния.

5. Все изменения состояния должны проходить через централизованную проверку. Нельзя обходить WIP-лимиты напрямую через DOM, drag-and-drop, localStorage, URL-import или внутренние helper-функции.

6. WIP-лимиты обязательны:
   - `S`: максимум 1 карточка.
   - `A`: максимум 3 карточки.
   - `B`, `C`, `D`: без лимита.

7. Публичная ссылка не должна хранить полный tier list в URL. Для публичного обмена нужен короткий идентификатор и бесплатное внешнее хранилище, например Firebase Hosting + Firestore, если пользователь явно одобрит такую архитектуру.

8. Проект не должен индексироваться поисковиками. Сохраняй `noindex` meta-теги в `index.html`, запрет в `robots.txt` и `X-Robots-Tag` в `_headers`.

9. `localStorage` можно использовать только для черновика текущего пользователя.

10. Перед добавлением новой функции или поведения сначала поищи существующую похожую реализацию в проекте через `rg`.

11. Следуй существующему стилю кода: обычный JavaScript, DOM API, простые функции, понятные сообщения пользователю.

12. Не используй destructive git-команды вроде `git reset`, `git checkout` или `git restore` для отката своих изменений. Откатывай только свои правки точечным редактированием.

13. Не делай git push, не создавай pull request и не устанавливай зависимости без явного разрешения пользователя.

14. Отвечай пользователю коротко и просто. Технические детали добавляй только если пользователь попросил.

15. Не записывай ничего в память или memory-файлы без явного разрешения пользователя.
